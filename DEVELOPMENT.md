# How *Realm* Was Built — A Development Diary

This document captures how the game in this folder (`kingdom.py`) was developed,
following the brief and method in **`kingdom-loop.md`**. It is written in two
layers:

- **Part 1 — The story** (plain language, no code needed): what the process was,
  how many loops it took, and what happened in each.
- **Part 2 — The technical appendix** (for developers): the specific code, tests,
  bugs, and design decisions in each cycle.

The honest version is told here — including the false starts and the things that
broke — because that *is* the process. A clean highlight reel would hide the part
that actually makes the method work.

---

# Part 1 — The Story

## What we were asked to build
A self-contained, text-based fantasy-kingdom management game — a tiny *Crusader
Kings* / *Football Manager*. You play a hereditary ruler: set taxes, build the
realm, marry off heirs, do diplomacy, wage war, and keep your dynasty on the
throne. The hard rules from the brief: **everything is generated by the program
itself** (every character, event, price, war outcome), **no internet, no outside
data, ever**, and **one simple tool-free tech stack** (we used plain Python).

## The method: a self-correcting loop
The brief didn't just ask for a game — it asked for a *way of working*. Instead
of writing the whole thing once and hoping, we ran a repeating loop:

> **build a small piece → play-test it automatically → look for what's weak or
> broken → write down what we learned → decide the next piece → repeat.**

Crucially, six plain-text files act as the project's **memory** (the spec, the
architecture notes, the task list, the test plan, the play-test review, and the
lessons log). They're kept up to date every loop, so the work could be picked up
cold by anyone — or any future session — from the files alone.

The loop only stops when a strict checklist is fully met: the game must survive
many full playthroughs with different random starts and *never* crash or break
its own rules, every promised feature must have an automatic test that proves it
works, every player choice must visibly *do something*, the to-do list must be
empty, and there must be a README.

## How many loops it took: **7**
The core game was built in **one continuous autonomous session across 5 loops**;
then **playing a full reign** (a later, hands-on session) exposed two deeper
flaws and reopened the loop for **2 more** — 7 in total. Here's what each was for:

| Loop | Theme | What changed |
|------|-------|--------------|
| **1** | Build the whole skeleton + the "proof" tests | The entire game, plus 7 automatic tests, plus all the memory files. |
| **2** | Fix repetition | Stop the same random event happening twice in a row. |
| **3** | Fix a grind | Stop endless back-to-back wars against the same neighbour. |
| **4** | Make consequences bite | Make debt actually hurt; add a final "score" for a reign. |
| **5** | Final review + a safety catch | A reviewer spotted an under-tested danger zone; we hardened it and proved it safe. |
| **6** | Fix a dynastic dead-end | A young heir who inherited childless could never continue the line; added royal marriage and heir protection. |
| **7** | Fix late-game boredom | A perfect realm became a frictionless do-nothing; made stability cost upkeep and rivals grow over time. |

> **The most important lesson of the whole project lives in loops 6–7:** eleven
> automatic tests all passed, but *playing one complete reign from start to
> finish* revealed problems no per-turn test could see. A simulation isn't done
> when the unit tests pass — it's done when you've played the whole arc.

## The loop-by-loop story

**Loop 1 — build it all, and build the proof at the same time.**
We wrote the complete game in a single file: a realm with gold, population,
prosperity, stability and legitimacy; a ruler with a family; three neighbouring
kingdoms; a yearly turn that runs five steps (money → a random event → your
decree → wars → people age and rulers can die). Alongside it we wrote a
*self-play harness*: a robot that plays the game making random choices, used to
shake out crashes, and seven automatic tests — one for each promise in the spec.

The very first robot run survived 10 full reigns with no crashes. But one test
failed — and the interesting part is **the test was wrong, not the game.** Our
"every choice must do something" test was feeding choices into kingdoms that had
*already collapsed to zero*, where a "lose population" choice naturally can't
lose any more. We fixed the *test* to judge choices against a healthy kingdom,
which is what the rule actually means. We also caught the game auto-maxing
"prosperity" to 100 and ignoring it, and a cosmetic bug where a name came out as
"Mor var" with a stray space. All fixed.

**Loop 2 — variety.** Watching a play-test, the same "peasants riot" event fired
three times in eight years. We made the game never pick the same event two years
running, and added a test that checks 450 events for back-to-back repeats.

**Loop 3 — no war grind.** Because winning a war lowered relations, the robot
kept re-declaring war on the same neighbour every single year. We added a
two-year "cooldown" after any war before you can fight that neighbour again, and
tested it.

**Loop 4 — make it matter.** The play-tests showed kingdoms ending deeply in
debt (thousands of gold owed) while staying perfectly stable — debt was
toothless. We made the penalty grow with how deep the debt is. We also added a
**prestige score** shown at the end, so a reign has a final verdict. Both got
tests.

**Loop 5 — the review that mattered.** Everything passed, so we asked a stronger
reviewer model to check our work before declaring victory. It found a real blind
spot: our crash-tests mostly ran *short* reigns, where a ruler dying and an heir
taking over is **rare** — so the most complicated, most dangerous part of the
code (succession) had barely been tested under our own safety checks. We
tightened the safety check into a precise rule ("game over means: the ruler is
dead and no heir exists") and ran a much longer, succession-heavy stress test:
**100 reigns of up to 60 years each**, checking every rule after every turn.
Many of those reigns *did* end in a dynasty dying out — and nothing broke. Only
then did we stop.

**Loops 6 & 7 — what playing the whole game taught us.** After the build was
"done," we sat down and played a full 72-year reign as a peaceful, just king. Two
things went wrong that no per-turn test had flagged. First, the **dynasty hit a
dead-end**: an heir who took the throne too young to be married could never have
children — and the game gave no way to fix it, so a thriving kingdom simply went
extinct. Loop 6 added a "take a spouse" choice for the ruler and made it
impossible to accidentally marry away your last heir. Second, the **late game got
boring**: once a realm was perfect, every year was identical and no choice
mattered. Loop 7 made a great realm *cost effort to maintain* (contentment slowly
fades unless you keep investing) and made rival kingdoms grow stronger over the
decades, so resting on your laurels is dangerous. With those in, the same dynasty
that died at year 72 now survives unbroken for 200 years when its rulers marry.

## Where it ended up
The game is finished against the brief: it survives 100+ varied playthroughs
without crashing or breaking a rule, all 11 promise-tests pass, every choice
visibly changes the world, the to-do list is empty, and there's a README. You
can play it with `python kingdom.py`.

A real 5-year play-test (as "Corgar the Just" of Lorhelm) showed it works as a
*game*, not just as code: buying off two hostile neighbours with gifts turned
enemies into friends and kept the realm at peace, mercy during a plague cost
lives but won goodwill, and prosperity climbed every year a generous king let
the harvest stay with his people.

# Part 2 — Technical Appendix

## Stack & shape
- **Python 3 standard library only.** No dependencies, no network, no datasets.
- **Two source files:** `kingdom.py` (engine + CLI + self-play harness, one
  module on purpose — split deferred until a second front-end needs the engine
  alone) and `test_kingdom.py` (acceptance tests as plain `assert`s, no
  framework).
- **Six memory artifacts** per the brief: `spec.md`, `architecture.md`,
  `tasks.md`, `test_plan.md`, `ux_review.md`, `lessons_learned.md`, plus a
  promoted `PROJECT_RULES.md` and this `DEVELOPMENT.md`.
- **One RNG, threaded through `Game.rng`** — a seed fully determines a
  playthrough, which is what makes the harness reproducible and bugs repeatable.

## The engine in one paragraph
`process_turn(game, choose)` runs five phases: `phase_economy` (income/upkeep →
treasury; tax & debt → stability; prosperity chases stability; population
drifts), an **event** (`EVENTS[i](game) -> (prompt, [(label, action)])`), a
**decree** (`decree_options(game)`, built from current state so only valid
actions appear), `phase_war` (resolve wars + possible invasions), and
`phase_aging` (age all; possible birth; mortality; ruler death →
`_kill_ruler` → `_choose_heir`: child → sibling → crisis). The only seam between
engine and UI is `choose(prompt, options) -> index`: the CLI passes a human
prompter, the harness passes a random picker.

## Cycle-by-cycle technical log

### Cycle 1 — scaffold + executable acceptance criteria
**Built:** full data model (`Person`, `Neighbor`, `Realm`, `Game` dataclasses),
the five-phase turn, all five systems (treasury, succession, war, diplomacy,
events), interactive CLI (`play`, `_ask_human`, `_print_status`), self-play
harness (`self_play`, `run_harness`), invariant checker (`check_invariants`),
and acceptance tests **A1–A7**. Authored all six memory artifacts + README.

**Results / bugs:**
- First harness run: 10 seeds × 30 turns, zero invariant violations.
- Acceptance suite went **6/7 → 7/7** after fixing a **test-fairness bug**: the
  `test_no_dead_choices` check applied each option to *collapsed* end-states
  (stats already at the floor), so a "−population" option legitimately produced
  no delta and was falsely flagged "dead." It also had a closure bug — the reset
  copy didn't rebind the option callbacks. Rewrote it to test every option
  against **healthy** states (the correct meaning of "no dead choice exists by
  design"); now 167 option-checks (104 event + 63 decree).
- **Game bug:** prosperity drifted +1 every stable turn and pinned at 100, going
  dead as a variable. Fixed so **prosperity chases stability** (≤+2/turn up,
  −1/turn down toward the stability value), so 100 prosperity requires sustained
  ~100 stability.
- **Cosmetic bug:** a `_MALE` syllable was `" var"` (leading space) → "Mor var".
  Removed the space.
- Also resolved two Pyright diagnostics (a `None` initial `ruler` — fixed by
  building the ruler before the `Game`; an unused lambda param).

### Cycle 2 — c1-a: no back-to-back events
Added `Game.last_event`; the event draw is `[e for e in EVENTS if e.__name__ !=
g.last_event] or EVENTS`. New test **A8**: 450 events across 15 seeds, zero
back-to-back repeats.

### Cycle 3 — c1-b: war cooldown
Added `Neighbor.cooldown`, set to `2` whenever a war ends (win or lose),
decremented each turn in `phase_economy`. Gates both the player's "declare war"
decree and AI invasions. New test **A9**.

### Cycle 4 — c1-d debt scaling + c1-c prestige score
- Debt penalty changed from a flat `−3` stability to `min(10, 3 +
  (-treasury)//1000)` — a modest overdraft is survivable, ruinous debt is not.
  Evidence: the same deep-debt seeds that previously held stability ~90 now
  collapse to ~19–27. New test **A10**.
- Added `prestige_score(game)` (years reigned, legitimacy, stability,
  prosperity, surplus, population), shown at game over and abdication. New test
  **A11**.

### Cycle 5 — final review + hardening (advisor-driven)
A stronger reviewer flagged a coverage gap sitting directly on the "zero
invariant violations" exit condition: invariants had only been asserted in 30-
and 40-turn harness runs, where succession is rare; the 400-seed extinction
*probe* skipped `check_invariants` entirely. So the most complex path —
succession/game-over — had barely run under the invariant contract at the rate
it occurs. Also the game-over branch of `check_invariants` was near-vacuous.

**Fixes:**
- Strengthened the game-over invariant into a real contract: `over ⟹
  over_reason set ∧ ruler dead ∧ _choose_heir() is None`.
- Ran the succession-heavy regime through invariants every turn:
  **`python kingdom.py --selfplay 100 60` → PASS.** Many seeds hit GAME OVER;
  the per-turn invariant held throughout. This validates the subtle cascade in
  `phase_aging` where an heir who ascends *while the captured `[ruler]+family`
  loop is still iterating* could take a second mortality roll — it never leaves
  `ruler.alive == False` with `over == False`.

### Cycle 6 — dynastic dead-end (c6-b), found by playing a full reign
`_kill_ruler` only granted a spouse at `heir.age >= 18`, no decree married the
*ruler*, and `decree_options` would marry out an only child — so a young
childless heir was an unrecoverable dead-end. Fixes: added a "Take a spouse"
decree (ruler ≥16, unmarried); lowered the ascension threshold 18→16; gated the
alliance "Wed" decree behind `len(heirs) >= 2`; cut "hold court" from +2 to +1
(also relevant to c7). Tests **A12, A13, A14**.

### Cycle 7 — late-game stasis (c6-a)
A peaceful realm pinned at 100/100 with identical turns. Fixes in
`phase_economy`: `if r.stability > 50: r.stability -= 1` (excellence erodes
without upkeep; passive equilibrium ≈50), and neighbours gain strength
(`if rng < 0.15: n.strength += randint(10,40)`) so a static army eventually
invites trouble. Test **A15**; the harness confirms random play no longer sits
at 100. *Residual (logged, unfixed):* gold/population grow unbounded past ~100
turns — outside the spec's ~30-turn envelope (YAGNI).

## Final verification (what "done" rests on)
- **Crash/invariant:** `100 seeds × 60 turns` + `50 × 30` + `25 × 40`, zero
  crashes / zero invariant violations (the brief asks for 10 in a row).
- **Acceptance tests:** A1–A11 all green (`python test_kingdom.py`).
- **No dead choices:** 167 option-checks against healthy states.
- **Reachability:** natural dynasty-extinction occurs 166/400 seeds over 60
  turns — the crisis path is live code, not dead code.
- **Playable:** `python kingdom.py` runs; a scripted 5-year reign demonstrated
  meaningful, consequential play.

## Reusable patterns (promoted to `PROJECT_RULES.md`)
1. Make acceptance criteria **executable** — the system tests itself; the
   headline test is a random-agent self-play harness that asserts invariants
   after every step.
2. "No dead choices" = clone a **healthy** state, apply each option, assert a
   delta (testing collapsed states gives false positives).
3. When a self-check fails, **first ask whether the check is fair** before
   changing the system under test (Cycle 1's whole lesson).
4. One seeded RNG threaded through the engine → reproducible runs and repros.
5. **Clamp on write, assert on read** — bound every value at mutation *and*
   re-assert it centrally each turn.
6. One UI seam (`choose(prompt, options) -> index`) so the engine never knows if
   it's talking to a human or a test harness.
