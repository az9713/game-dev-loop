Build me a self-contained fantasy-kingdom management game — a text-based
Crusader Kings / Football Manager with no licensed engine and NO external data.
I'm the ruler: decrees, treasury, marriages, wars, succession. All content —
characters, events, economy, war outcomes — is procedurally generated by the
simulation itself. No network calls, no APIs, no downloaded datasets ever.

Run this through the /LFG compound-engineering flow (brainstorm → plan → build →
review → improve), and KEEP LOOPING until the exit conditions below are met.
After each cycle, write yourself the next instruction and start the next cycle
without waiting for me. If I'm asleep, keep going.

═══════════════════════════════════════════════════════════
ARTIFACTS — create and maintain these files. They are the project's memory.
═══════════════════════════════════════════════════════════

spec.md — WHAT we're building and HOW we'll know it's done.
  Contains: product concept, the player's core loop (a single turn), the list
  of systems (treasury, succession, war, diplomacy, events), and the
  acceptance criteria as a checklist of objective, self-verifiable tests.
  Success criteria for THIS doc: every system has at least one acceptance
  test phrased so the simulation itself can pass/fail it (no human judgment).
  No vague goals ("make it fun") without a mechanical proxy.

architecture.md — HOW the code is shaped.
  Contains: module boundaries (sim core / state model / RNG / event system /
  CLI), the game-state data structure, how a turn is processed end-to-end, and
  the invariants that must never be violated (e.g. treasury is an integer,
  succession always resolves to exactly one heir or a defined crisis state).
  Success criteria: a new turn's data flow is traceable in one read; every
  invariant names the code that enforces it.

tasks.md — the ordered backlog.
  Contains: tasks small enough to finish in one cycle, each with its own
  done-test. Mark [ ] / [x]. Append new tasks as the review step finds gaps.
  Success criteria: the next actionable task is always unambiguous; no task is
  "improve X" without a concrete check.

test_plan.md — how each system is exercised.
  The headline test is a self-play harness: advance a full reign (~30 turns)
  with scripted/random choices and assert no invariant breaks and no crash.
  Success criteria: running the harness is one command and prints PASS/FAIL.

ux_review.md — what playing it actually feels like.
  After self-playing, log: which choices had no visible consequence, which
  events repeated too often, where the player had nothing meaningful to decide.
  Success criteria: every entry names a concrete mechanical fix, not a vibe.

lessons_learned.md — the COMPOUND step (do this every cycle).
  Per cycle append: work done / evaluation / gaps found / next instruction /
  reusable pattern. When a pattern is reusable across future projects, also
  promote it to PROJECT_RULES.md.

═══════════════════════════════════════════════════════════
LOOP RULES — one cycle:
═══════════════════════════════════════════════════════════
1. Pick the next [ ] task from tasks.md (or, if reviewing, the top gap).
2. Implement the smallest increment that completes it.
3. Run test_plan.md's self-play harness. Inspect the output as a designer
   would, not just "did it crash" — look for shallow, missing, or unfair
   mechanics and log them to ux_review.md.
4. Diagnose: what's broken, missing, overcomplicated, or unverified?
5. Update spec.md / architecture.md / tasks.md to reflect reality.
6. Append the cycle to lessons_learned.md and write the next instruction.
7. Start the next cycle.

═══════════════════════════════════════════════════════════
EXIT CONDITIONS — stop looping only when ALL are true:
═══════════════════════════════════════════════════════════
✓ Self-play harness runs a full ~30-turn reign with zero crashes and zero
  invariant violations, 10 times in a row with different RNG seeds.
✓ Every acceptance test in spec.md passes.
✓ Every system (treasury, succession, war, diplomacy, events) has a visible
  mechanical consequence verified in ux_review.md — no dead choices remain.
✓ tasks.md has no open [ ] items and the review step finds no new gap.
✓ A README explains how to run it and play it.
Also stop and summarize if you hit the same failing test 3 cycles running
(you're stuck — write up what you tried and where it's blocked).
